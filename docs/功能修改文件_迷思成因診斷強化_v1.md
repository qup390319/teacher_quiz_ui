# 迷思概念診斷系統 — 功能修改文件 v1.0

> 日期：2026-04-17
> 範圍：學生端對話診斷 + 教師端診斷報告
> 目標：強化迷思成因辨識、優化追問歸類、提升教師回饋品質

---

## 一、修改總覽

| 項目 | 現況 | 目標 |
|------|------|------|
| 迷思成因分析 | 僅有 `misconceptionSource` 欄位（一句話描述），未結構化分類 | 建立「成因類型 × 問題類型」雙軸診斷矩陣，每個迷思都有明確的成因歸類與教學對策 |
| 追問策略 | 固定對話腳本，無依據學生回答動態分類 | 追問結束後依對話內容歸類為「觀察型／定義型／解釋型」問題，並記錄判斷依據 |
| 教師回饋 | 僅顯示「答錯比例」及「迷思名稱」 | 提供學生思考邏輯摘要、錯誤成因分析、問題類型分布圖、具體教學建議 |

## 二、資料結構修改

### 2.1 題目選項擴充（`quizData.js`）

每個選項新增 `causeAnalysis` 物件，建立「學生選擇此選項 → 背後可能的錯誤成因」對照表。

```javascript
// ── 修改前 ──
{
  tag: 'A',
  content: '鐵椅的溫度本來就比木頭低。',
  diagnosis: 'M01-1'
}

// ── 修改後 ──
{
  tag: 'A',
  content: '鐵椅的溫度本來就比木頭低。',
  diagnosis: 'M01-1',
  causeAnalysis: {
    // ── 成因分類 ──
    causeType: 'daily-experience',         // 成因類型（見 2.3 成因類型列舉）
    causeLabel: '日常觸覺經驗的錯誤歸因',    // 成因名稱（給老師看）
    causeDetail: '學生將「摸起來冰冰的」等同於「溫度較低」，'
              + '未區分「觸覺感受」與「實際溫度」的差異。'
              + '此誤解通常來自日常生活中頻繁接觸金屬表面的經驗。',
    
    // ── 問題類型歸類 ──
    problemType: 'observation',             // 觀察型 | 定義型 | 解釋型
    problemTypeReason: '學生將主觀觸覺感受（冰冰的）當作客觀觀察（溫度低），'
                     + '核心困難在於「感覺到什麼」與「實際是什麼」的區分。',

    // ── 蘇格拉底式追問方向 ──
    followUpStrategy: {
      targetInsight: '觸覺感受 ≠ 實際溫度，需要用溫度計測量才能確定',
      conflictScenario: '夏天摸同一張鐵椅會覺得燙，但鐵椅不可能冬天冷夏天熱',
      guidingQuestions: [
        '如果拿溫度計分別量鐵椅和木椅，你覺得讀數會一樣嗎？',
        '夏天時摸同一張鐵椅，你覺得會是什麼感覺？為什麼？',
        '那你覺得是什麼因素影響了我們的「觸覺感受」？'
      ]
    }
  }
}
```

## 三、追問策略設計

### 3.1 蘇格拉底式追問三輪架構

每一題的 AI 追問依以下框架設計，目的是**透過結構化對話確認學生的問題類型與迷思成因**：

```
┌─────────────────────────────────────────────────────┐
│ 第 1 輪：開放式探問（暴露想法）                          │
│ 目的：讓學生用自己的話解釋，暴露推理過程                    │
│ 範例：「你為什麼選這個答案？可以說說你的想法嗎？」            │
│                                                       │
│ AI 分析重點：                                           │
│  → 學生使用的關鍵詞彙（科學語言 or 日常語言？）              │
│  → 推理方向（觀察層面 or 定義層面 or 因果層面？）            │
│  → 信心程度（肯定 or 猶豫？）                             │
├─────────────────────────────────────────────────────┤
│ 第 2 輪：針對性追問（製造認知衝突）                        │
│ 目的：用反例或比較挑戰學生的想法，測試迷思穩固程度             │
│                                                       │
│ 依問題類型選擇追問策略：                                  │
│                                                       │
│  觀察型 → 「如果我們用儀器測量，你覺得結果會是...？」        │
│          引導學生區分感覺與測量                            │
│                                                       │
│  定義型 → 「你覺得○○是什麼意思？跟△△有什麼不一樣？」       │
│          引導學生釐清概念邊界                              │
│                                                       │
│  解釋型 → 「如果你的說法是對的，那為什麼...？」              │
│          用反例挑戰因果推論                                │
│                                                       │
│ AI 分析重點：                                           │
│  → 學生是否開始動搖？                                    │
│  → 修正方向是否正確？                                    │
│  → 能否自行連結到正確概念？                                │
├─────────────────────────────────────────────────────┤
│ 第 3 輪：引導收斂（協助建構）                              │
│ 目的：確認概念修正程度，生成最終診斷                        │
│ 範例：「根據你剛才的思考，現在你會怎麼重新解釋這個現象？」      │
│                                                       │
│ AI 分析重點：                                           │
│  → 最終歸類：觀察型 / 定義型 / 解釋型                      │
│  → 成因判定：日常經驗 / 語言混淆 / 過度類推 / ...           │
│  → 概念修正程度：none / partial / complete                │
└─────────────────────────────────────────────────────┘
```

### 3.2 追問確保診斷的邏輯

追問如何確保診斷的準確性（對應「為什麼對話可以確保診斷」）：

| 問題 | 僅靠選擇題的風險 | 追問如何解決 |
|------|----------------|-----------|
| 猜對了 | 學生答對但推理錯誤，系統無法偵測 | 追問「為什麼？」暴露推理，發現「猜對」或「背答案」 |
| 不確定的迷思 | 選項只能對應預設迷思，不知學生真正想法 | 開放式對話可發現選項以外的非典型迷思 |
| 迷思成因不明 | 同一個錯誤選項，不同學生的成因可能不同 | 對話中的用詞和邏輯可區分成因類型 |
| 概念是否修正 | 無法知道學生是否「被說服」 | 第 3 輪收斂追問可確認概念是否真的改變 |

**診斷依據記錄**：每一輪對話結束後，系統記錄 AI 的判斷依據：

```javascript
diagnosticTrace: [
  {
    round: 1,
    studentKeywords: ['冰冰的', '比較冷'],
    signalDetected: '使用感覺詞彙描述溫度，未使用科學語言',
    preliminaryType: 'observation',
    preliminaryCause: 'daily-experience',
    confidence: 'medium'
  },
  {
    round: 2,
    studentKeywords: ['可能跟材質有關', '但摸起來真的冰'],
    signalDetected: '開始動搖但仍傾向感覺判斷',
    refinedType: 'observation',
    refinedCause: 'daily-experience',
    conceptShift: 'partial',
    confidence: 'high'
  },
  {
    round: 3,
    studentKeywords: ['導熱', '散熱比較快'],
    signalDetected: '能說出科學關鍵詞但邏輯鏈不完整',
    finalType: 'observation',
    finalCause: 'daily-experience',
    conceptShift: 'partial',
    confidence: 'high'
  }
]
```

### 3.3 各問題類型的追問範例（以「熱傳導」Q1 為例）

#### 觀察型追問路線（學生選 A：鐵的溫度比木頭低）

```
AI：你選了「鐵椅的溫度本來就比木頭低」，可以說說你為什麼這樣覺得嗎？

學生：因為冬天摸鐵椅冰冰的，木椅不會啊。

  → AI 判斷：學生依據觸覺經驗，把感覺等同觀察結果
  → 問題類型信號：觀察型（感覺 = 觀察）
  → 成因信號：日常經驗干擾

AI：有趣的想法！那我想問你，如果我們拿溫度計分別量鐵椅和木椅的表面溫度，
   你覺得讀數會是一樣的還是不一樣？

學生：應該不一樣吧...鐵的應該比較低？

  → AI 判斷：學生仍堅持，但語氣轉為不確定
  → 概念修正程度：微動搖

AI：其實在同一個環境中放了一整晚的話，溫度計顯示的讀數會是一樣的喔！
   那你覺得，既然溫度一樣，為什麼摸起來的感覺會不同呢？

學生：嗯...可能是因為鐵比較會傳熱？把我手上的熱帶走比較快？

  → AI 最終判斷：
  → 問題類型 = 觀察型（核心困難已釐清）
  → 成因 = 日常經驗干擾
  → 概念修正 = partial（能說出「傳熱」但邏輯鏈仍需補強）
```

#### 定義型追問路線（假設學生混淆「熱」和「溫度」）

```
AI：你提到「鐵椅比較冷所以溫度比較低」，我想請問你覺得「溫度」是什麼意思？
   跟「熱」有什麼不一樣嗎？

學生：溫度就是熱不熱啊，一樣的意思。

  → AI 判斷：概念混淆，將「溫度」和「熱」視為同義詞
  → 問題類型信號：定義型
  → 成因信號：語言概念混淆

AI：你覺得「一杯 80°C 的水」跟「一鍋 80°C 的水」，溫度一樣嗎？熱量一樣嗎？

學生：溫度一樣...熱量...嗯，鍋子比較多應該比較熱？

  → AI 判斷：學生開始意識到溫度和熱不完全相同
  → 概念修正程度：部分動搖

AI：沒錯！溫度是「程度」，熱量是「總量」，就像「甜度」和「總糖量」的差別。
   用這個理解重新想想，冬天鐵椅的溫度真的比木椅低嗎？

  → AI 最終判斷：
  → 問題類型 = 定義型（混淆溫度/熱）
  → 成因 = 語言概念混淆
  → 概念修正 = partial
```

#### 解釋型追問路線（學生選 D：「冷」在鐵中傳播比木頭快）

此例以同一題（Q1 熱傳導）的選項 D 為情境。學生能正確觀察到「鐵摸起來比較冰」，也知道跟「傳播速度」有關，但在因果機制上出錯——認為是「冷」這種東西在鐵中傳播得比較快。

```
AI：你選了「冷在鐵中傳播的速度比木頭快」，可以說說你怎麼理解「冷的傳播」嗎？

學生：就是冷會從鐵椅傳到我的手上啊，鐵傳得比較快所以很快就覺得冰了。

  → AI 判斷：學生的觀察正確（鐵椅確實較快感覺到冰），
             也知道跟傳播速率有關，但將「冷」當成一種可傳播的實體。
  → 問題類型信號：解釋型（觀察與名詞大致正確，但因果機制有誤）
  → 成因信號：直覺思維 + 語言概念混淆（「冷的傳播」是日常說法的誤導）

AI：你提到「冷從鐵椅傳到手上」，那我想問你——
   如果冷氣房裡有一張鐵椅和一張木椅都放了一整晚，
   你覺得「冷」是從哪裡來的？是鐵椅自己產生的嗎？

學生：嗯...應該不是鐵椅產生的，是整個房間都冷，然後鐵傳冷比較快？

  → AI 判斷：學生否認鐵椅自行產冷（正確），但仍堅持「傳冷」的說法。
             核心迷思是把「冷」當成一種能被傳遞的物質或能量。
  → 概念修正程度：微動搖（知道冷不是鐵產生的，但因果機制未修正）

AI：其實在科學上，「冷」不是一種會傳播的東西喔。真正在傳播的是「熱」——
   你手上的熱透過鐵快速傳走了，所以你「失去熱」的速度很快，就覺得冰。
   換句話說，不是「冷傳過來」，而是「熱被帶走」。
   你覺得這兩種說法有什麼差別嗎？

學生：喔...所以是我的熱跑到鐵上面去，不是冷跑過來？
     那其實方向是反過來的！

  → AI 最終判斷：
  → 問題類型 = 解釋型（因果機制方向倒置：「冷傳來」vs「熱傳走」）
  → 成因 = 因果倒置 + 語言概念混淆
     · 因果倒置：把「熱流失」的結果（感覺冷）當成了原因（冷傳過來）
     · 語言混淆：日常語言「冷冰冰的感覺傳過來」強化了冷是實體的誤解
  → 概念修正 = partial → complete
    （學生主動說出「方向反過來」，表示因果關係已修正）

  → thinkingChain:
    前提：摸鐵椅很快就覺得冰（正確觀察）
    推論：一定是「冷」從鐵椅傳到手上（錯誤因果方向）
    結論：冷在鐵中傳播速度比木頭快（錯誤機制）
    缺失環節：未理解「熱從手傳到鐵」才是實際機制，
             「感覺冷」是熱流失的結果而非原因

  → diagnosticConfidence: high
    學生清楚表達了「冷傳過來」的因果模型，
    且在引導後能自行說出「方向反過來」，修正軌跡明確。
```

**三種問題類型追問策略的關鍵差異對照：**

| | 觀察型 | 定義型 | 解釋型 |
|---|---|---|---|
| 學生的核心困難 | 把感覺當觀察 | 把日常語義當科學定義 | 因果方向或機制搞錯 |
| 第 1 輪追問重點 | 「你是怎麼知道的？」 | 「你覺得○○是什麼意思？」 | 「你覺得為什麼會這樣？」 |
| 第 2 輪衝突策略 | 引入測量工具對比感覺 | 用極端案例測試定義邊界 | 用反例挑戰因果邏輯 |
| 第 3 輪收斂方向 | 區分「感覺」vs「測量」 | 釐清概念 A ≠ 概念 B | 重建正確因果方向 |
| 修正成功的標誌 | 學生提到「要用儀器確認」 | 學生能說出兩概念的差異 | 學生能重述正確的因果關係 |

## 八、後端架構設計

### 8.1 技術選型

| 層級 | 技術 | 理由 |
|------|------|------|
| 後端框架 | **Node.js + Express** | 與前端同語言（JavaScript），團隊學習成本最低 |
| 資料庫 | **PostgreSQL** | 關聯式資料適合師生、班級、作答紀錄等結構化資料 |
| ORM | **Prisma** | 類型安全、自動遷移、Schema 即文件 |
| 驗證 | **JWT + bcrypt** | 輕量級帳號系統，支援教師/學生角色 |
| LLM 接口 | **OpenAI-compatible SDK**（見第九章） | 統一介面，可切換不同 LLM Provider |
| 即時通訊 | **Server-Sent Events (SSE)** | AI 追問對話的串流回覆，比 WebSocket 簡單 |
| 環境設定 | **dotenv (.env)** | 所有敏感設定皆由環境變數管理 |

## 九、LLM 抽象層設計

### 9.1 架構：Provider Pattern

系統採用 **Provider Pattern** 建立 LLM 抽象層，使得切換不同 LLM 服務只需要修改 `.env` 設定，不需要改動任何業務邏輯程式碼。

```
┌─────────────────────────────────────────────────┐
│              業務邏輯層                           │
│  (diagnosis.js / promptBuilder.js / followUp.js) │
│                                                 │
│       只呼叫 llmProvider.chat(messages)          │
│       不需要知道底層用哪個 LLM                     │
└───────────────────┬─────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│            LLMProvider 抽象介面                   │
│                                                 │
│  chat(messages, options)  → string              │
│  chatStream(messages, options) → AsyncIterator  │
│  healthCheck() → boolean                        │
│  getModelInfo() → { provider, model, status }   │
└───────┬──────────┬──────────┬───────────────────┘
        │          │          │
        ▼          ▼          ▼
┌──────────┐ ┌──────────┐ ┌──────────────┐
│ VLLMProv │ │ OpenAIPr │ │ AnthropicPr  │
│ ider     │ │ ovider   │ │ ovider       │
│          │ │          │ │              │
│ 目前使用  │ │ 預留擴充  │ │ 預留擴充     │
└──────────┘ └──────────┘ └──────────────┘
```

### 9.2 抽象介面定義

```javascript
// backend/src/services/llm/LLMProvider.js

/**
 * LLM Provider 抽象基底類別
 * 所有 Provider 實作都必須繼承此類別
 */
class LLMProvider {
  constructor(config) {
    this.baseUrl = config.baseUrl;
    this.apiKey = config.apiKey;
    this.model = config.model;
    this.defaultOptions = {
      maxTokens: config.maxTokens || 1024,
      temperature: config.temperature || 0.7,
    };
  }

  /**
   * 非串流對話（一次回傳完整回覆）
   * @param {Array} messages - [{ role: 'system'|'user'|'assistant', content: string }]
   * @param {Object} options - { maxTokens, temperature, ... }
   * @returns {Promise<string>} AI 回覆文字
   */
  async chat(messages, options = {}) {
    throw new Error('chat() must be implemented by subclass');
  }

  /**
   * 串流對話（逐 token 回傳，用於 SSE）
   * @param {Array} messages
   * @param {Object} options
   * @returns {AsyncGenerator<string>} 逐 token yield
   */
  async *chatStream(messages, options = {}) {
    throw new Error('chatStream() must be implemented by subclass');
  }

  /**
   * 健康檢查
   * @returns {Promise<{ healthy: boolean, latencyMs: number, model: string }>}
   */
  async healthCheck() {
    throw new Error('healthCheck() must be implemented by subclass');
  }

  /**
   * 取得模型資訊
   * @returns {{ provider: string, model: string, baseUrl: string }}
   */
  getModelInfo() {
    return {
      provider: this.constructor.name,
      model: this.model,
      baseUrl: this.baseUrl,
    };
  }
}
```

### 9.3 vLLM Provider 實作

```javascript
// backend/src/services/llm/VLLMProvider.js

const LLMProvider = require('./LLMProvider');

class VLLMProvider extends LLMProvider {
  /**
   * vLLM 使用 OpenAI-compatible API 格式
   * 因此可以直接用 OpenAI SDK 或原生 fetch 呼叫
   */

  async chat(messages, options = {}) {
    const response = await fetch(`${this.baseUrl}/chat/completions`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${this.apiKey}`,
      },
      body: JSON.stringify({
        model: this.model,
        messages,
        max_tokens: options.maxTokens || this.defaultOptions.maxTokens,
        temperature: options.temperature || this.defaultOptions.temperature,
        stream: false,
      }),
    });

    if (!response.ok) {
      const err = await response.text();
      throw new Error(`vLLM API error (${response.status}): ${err}`);
    }

    const data = await response.json();
    return data.choices[0].message.content;
  }

  async *chatStream(messages, options = {}) {
    const response = await fetch(`${this.baseUrl}/chat/completions`, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
        'Authorization': `Bearer ${this.apiKey}`,
      },
      body: JSON.stringify({
        model: this.model,
        messages,
        max_tokens: options.maxTokens || this.defaultOptions.maxTokens,
        temperature: options.temperature || this.defaultOptions.temperature,
        stream: true,
      }),
    });

    if (!response.ok) {
      const err = await response.text();
      throw new Error(`vLLM stream error (${response.status}): ${err}`);
    }

    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;

      buffer += decoder.decode(value, { stream: true });
      const lines = buffer.split('\n');
      buffer = lines.pop();  // 保留不完整的最後一行

      for (const line of lines) {
        if (line.startsWith('data: ') && line !== 'data: [DONE]') {
          try {
            const json = JSON.parse(line.slice(6));
            const token = json.choices?.[0]?.delta?.content;
            if (token) yield token;
          } catch (e) {
            // 忽略解析失敗的行
          }
        }
      }
    }
  }

  async healthCheck() {
    const start = Date.now();
    try {
      const response = await fetch(`${this.baseUrl}/models`, {
        headers: { 'Authorization': `Bearer ${this.apiKey}` },
        signal: AbortSignal.timeout(5000),
      });
      return {
        healthy: response.ok,
        latencyMs: Date.now() - start,
        model: this.model,
      };
    } catch (e) {
      return { healthy: false, latencyMs: Date.now() - start, model: this.model, error: e.message };
    }
  }
}
```

### 11.1 API 客戶端

```javascript
// frontend/src/api/client.js

const BASE_URL = import.meta.env.VITE_API_BASE_URL || 'http://localhost:4000/api';

async function apiFetch(path, options = {}) {
  const token = localStorage.getItem('token');
  const res = await fetch(`${BASE_URL}${path}`, {
    headers: {
      'Content-Type': 'application/json',
      ...(token && { 'Authorization': `Bearer ${token}` }),
      ...options.headers,
    },
    ...options,
  });
  if (!res.ok) throw new Error(`API error: ${res.status}`);
  return res.json();
}

// SSE 串流（用於 AI 追問）
function apiStream(path, body, onToken, onDone) {
  const token = localStorage.getItem('token');
  fetch(`${BASE_URL}${path}`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      ...(token && { 'Authorization': `Bearer ${token}` }),
    },
    body: JSON.stringify(body),
  }).then(async (res) => {
    const reader = res.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
      const { done, value } = await reader.read();
      if (done) break;
      buffer += decoder.decode(value, { stream: true });
      const lines = buffer.split('\n');
      buffer = lines.pop();
      for (const line of lines) {
        if (line.startsWith('data: ')) {
          const data = JSON.parse(line.slice(6));
          if (data.done) { onDone(data); }
          else if (data.token) { onToken(data.token); }
        }
      }
    }
  });
}

export { apiFetch, apiStream };
```

原有的 `AppContext.jsx` 從「靜態資料 + useState」改為「API 呼叫 + useEffect 同步」：

| 原本（靜態） | 改為（API） |
|------------|-----------|
| `useState(QUIZZES_DATA)` | `useEffect → apiFetch('/quizzes')` |
| `useState(CLASSES_DATA)` | `useEffect → apiFetch('/classes')` |

## 十四、設計原則（更新版）

1. **環境變數管理**：所有 config（API URL、密鑰、模型名稱）都透過 `.env` 管理，絕不寫死在程式碼中。
2. **LLM Provider 可切換**：透過 Provider Pattern 抽象，修改 `.env` 中的 `LLM_PROVIDER` 即可切換 vLLM / OpenAI / Anthropic，不需改程式碼。
3. **漸進式遷移**：前端可保留靜態資料作為 fallback，當後端 API 不可用時自動降級為 Demo 模式。
4. **教師確認機制**：`causeAnalysis` 中的成因對照，設計上需經教師審閱確認，系統提供預設值但非最終定論。
5. **漸進式揭露**：教師報告預設顯示摘要，點擊可展開詳細的思考邏輯和對話紀錄。
6. **不評判學生**：學生端報告中避免使用「錯誤」等負面詞彙，改用「你的想法」vs「科學的看法」。
7. **教師端為主要受眾**：成因分析、問題類型等專業資訊主要呈現在教師端，學生端僅提供友善的學習建議。
8. **SSE 串流體驗**：AI 追問採用 Server-Sent Events 逐 token 串流，學生端呈現「打字機效果」，提升對話即時感。
9. **診斷可追溯**：每一輪對話的 AI 判斷依據都記錄在 `diagnosticTrace`，教師可完整追溯「AI 為什麼做出這個診斷」。
